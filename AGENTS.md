# AGENTS.md — Annotate AI

## What This Is

A WordPress plugin that adds a visual annotation toolbar for AI-assisted design feedback. Admins click elements on the frontend, leave notes, and AI agents fetch and action them via REST API.

> **Installing the plugin from scratch?** [`SETUP_FOR_AGENTS.md`](SETUP_FOR_AGENTS.md) is a paste-ready prompt for AI coding agents covering install, build, the Application Password handoff, verification, and the API contract. This file (`AGENTS.md`) is the deeper reference once it's already installed.

## Design Decisions

### Frontend: TypeScript + React + @wordpress/components
The toolbar is a React component (`src/frontend/Toolbar.tsx`) that mounts into a dedicated root appended to `<body>`. Built with `@wordpress/scripts`. Loaded only for users with `manage_options`.

UI is built from WordPress core components wherever they apply (`Modal`, `Snackbar`, `Notice`, `Button`, `TextareaControl`) so the toolbar matches WordPress visual conventions and inherits a11y, theming, and behaviour as those components evolve. Custom CSS is intentionally minimal — positioning rules for the numbered pins, the "Submit N" pill, the floating-toggle fallback, and the admin-bar dashicon. All custom classes prefixed with `aai-`.

### Toolbar trigger: WordPress admin bar (with floating fallback)
The annotation-mode toggle is added as a node in the WordPress admin bar (`admin_bar_menu` hook) using a `dashicons-post-status` icon. React attaches its click handler to the rendered `<a>` and reflects active state by toggling an `is-active` class on the bar item.

If the admin bar is hidden for the current user (profile preference, theme filter, etc.), `is_admin_bar_showing()` is `false` and React falls back to a floating `<Button variant="primary">Annotate</Button>` parked bottom-right.

### Admin colors on the frontend
WordPress only sets `--wp-admin-theme-color` inside wp-admin. The plugin enqueues the user's chosen admin colors stylesheet on the frontend so wp-components renders with the same color scheme as wp-admin (Fresh, Coffee, Ectoplasm, etc.).

### Admin: TypeScript + @wordpress/components
The settings page (`src/admin/index.tsx`) uses React with WordPress components (`Card`, `CardHeader`, `CardBody`, `SelectControl`, `TextControl`, `Button`, `Notice`, `Spinner`, `__experimentalHStack`, `__experimentalVStack`). Built with `@wordpress/scripts`. TypeScript for type safety. Activation of the plugin is the on/off switch — there's no "enable" toggle.

### Styles: SCSS via @wordpress/base-styles
All design tokens come from `@wordpress/base-styles/colors` and `@wordpress/base-styles/variables`:
- Colors: `wp-colors.$gray-900`, `$alert-red`, `$alert-green`, etc. — `--wp-admin-theme-color` for the admin theme color (with `#3858e9` as the default fallback in `var()`).
- Spacing: `wp-vars.$grid-unit-05` … `$grid-unit-80` (4px–64px in 8px increments).
- Typography: `wp-vars.$font-size-medium`, `$font-family-mono`, `$font-line-height-x-small`.
- Radii: `wp-vars.$radius-block-ui`.
- Layout widths: `wp-vars.$content-width` (840px) for settings pages.
- Elevation: `wp-vars.$elevation-small` … `$elevation-large` for box shadows on floating UI.

SCSS is compiled by wp-scripts into `build/style-admin.css` and `build/style-frontend.css`. Avoid hardcoding hex values, px sizes, or shadow recipes.

### Annotation modal: direct manipulation + notes
Clicking an element opens a modal with two complementary input modes:
- **Tweak values** — pre-filled inputs for the properties most often changed during design refinement: `text` content (when the element has only text children), `font-size`, `color`, `background-color`. Each starts at the current computed value; only properties the user actually changes are sent.
- **Notes** — free-text field for everything that doesn't fit (layout moves, conditional visibility, "feels off" gut checks).

The submitted payload contains both the full `computed_styles` snapshot (context) and a `requested_changes` object with only the deltas (instructions). Plus optional `requested_text` and free-text `note`. This way the agent gets unambiguous targets for the easy 80% and natural-language guidance for the long tail.

### Persistence model depends on notification method
- **None** (agent pulls via REST API) — annotations persist *immediately* on Add. No separate "Send" step. Edits PATCH the server in place.
- **Webhook / Telegram** — annotations batch in component state until the user clicks **Send N annotations**, which POSTs to `/batch` and triggers a single notification per batch (so the agent isn't spammed per-annotation).

### Persistent pins across sessions
On every page load, the toolbar fetches all annotations for the current `page_url` (any status except `verified`/`resolved`) and renders pins in their current status colour: blue (open), yellow (in_progress, read-only), green (done — opens a review modal). Pin positions are recomputed on mount and on resize by re-running `document.querySelector(annotation.selector)` — pins for stale selectors (because the page was restructured) are silently dropped.

This is the QA loop: the designer leaves a pass of pins → agent works through them in the background → designer comes back later, sees the green ones, clicks each to review the agent's diff, hits "Looks good" or "Not quite" to verify or send back for revision.

The settings page shows the pull URL + an agent-instructions snippet (with copy buttons) whenever method is None, so users can quickly wire up Claude Code / Codex / etc.

### Storage: WordPress options (not a CPT)
Annotations are temporary design feedback stored in `wp_options` as a serialised array, capped at `MAX_ANNOTATIONS = 500` (oldest entries pruned). No CPT, no database tables, no migration complexity.

### No priority or category fields
Deliberately removed. The annotation is: element + note. The agent figures out the rest.

### Admin only (`manage_options`)
This is a development/QA tool. Only admins see the toolbar; all REST routes require `manage_options`.

### Author derived server-side
The annotation `user` field is always set from `wp_get_current_user()` in PHP — the client cannot supply it.

## Internationalization

Every user-facing string is wrapped in `__()` / `_x()` / `sprintf()` against the `annotate-ai` text domain.

- **PHP:** `__( 'string', 'annotate-ai' )` — text domain loaded via `load_plugin_textdomain` on `init`.
- **TypeScript/React:** `import { __, sprintf } from '@wordpress/i18n'`.
- **Translations enqueued for both bundles:** PHP calls `wp_set_script_translations()` for both `annotate-ai-admin` and `annotate-ai-toolbar`, pointing at `languages/`.

### Bundled languages

`.po`, `.mo`, and `.json` files are checked in for: `fr_FR`, `de_DE`, `es_ES`, `pt_BR`, `it_IT`, `ja`. PHP-side translations work as soon as the plugin is activated. The `.json` files (one per script handle, named with an md5 of the script path) are loaded at runtime by `wp_set_script_translations()`.

### Translation workflow

After modifying any user-facing string:

```bash
npm run build         # produce build/admin.js + build/frontend.js
npm run makepot       # extract strings → languages/annotate-ai.pot
# Update existing .po files (translations preserved, refs updated):
for po in languages/annotate-ai-*.po; do
    msgmerge --update --backup=none --no-fuzzy-matching "$po" languages/annotate-ai.pot
done
npm run makemo        # compile .po → .mo (PHP)
npm run makejson      # generate .json files (JS)
# Or once everything is in place: npm run i18n
```

`makepot` scans `build/*.js` (excluding `node_modules` and `src`), so the build must be run first or string references will be missing. `make-json` looks up JS translations by `md5(<relative-script-path>)`, which is why the built JS file paths must appear in the `.po` `#:` comments.

## Connecting an Agent

The REST API is the universal connector.

| Scenario | How to connect |
|----------|---------------|
| Remote agent (OpenClaw, etc.) | `GET https://your-site.com/wp-json/annotate-ai/v1/annotations?status=open` |
| Claude Code (remote site) | Same — REST API remotely |
| Claude Code (localhost) | `GET http://localhost:{port}/wp-json/annotate-ai/v1/annotations?status=open` |
| Codex (localhost) | Same as Claude Code |

### Agent Workflow

Annotations move through a status flow: `open` → `in_progress` → `done` → `verified` (or back to `open` if the human says "not quite"). Pins on the page colour-code the status.

1. **Fetch open annotations:**
   ```
   GET /annotate-ai/v1/annotations?status=open
   ```
2. **Claim one** so the user sees it's being worked on (pin turns yellow):
   ```
   PATCH /annotate-ai/v1/annotations/{id}
   { "status": "in_progress" }
   ```
3. **Read the payload:**
   - `selector`, `element_tag`, `element_text` — what was annotated
   - `computed_styles` — full snapshot of the element's relevant computed CSS (context)
   - `requested_changes` — structured deltas, each shaped `{"value": "24px", "preset": "x-large"}`. The `preset` slug, when present, is a `theme.json` slug — write to it (`fontSize: "x-large"` on a block, or `theme.json.styles.elements.h2.typography.fontSize`) rather than emitting raw CSS.
   - `requested_text` — new text content (text-only elements only)
   - `breakpoint` — `"all"` | `"mobile"` | `"tablet"` | `"desktop"`. If not `"all"`, the user is flagging an issue specific to that viewport — test the fix there, not just at the size in `viewport`.
   - `note` — free-text guidance for anything the structured fields couldn't capture
4. **Apply the fix** via the WordPress REST API: `theme.json` global styles, block attributes inside `post_content`, customiser CSS, theme files.
5. **Mark done** with a structured change-log so the human can verify (pin turns green, click opens a review modal):
   ```
   PATCH /annotate-ai/v1/annotations/{id}
   {
     "status": "done",
     "resolution_note": "Bumped hero size and tightened header spacing.",
     "changes": [
       { "file": "theme.json", "path": "styles.elements.h2.typography.fontSize", "old": "18px", "new": "24px" },
       { "file": "wp_posts/42", "field": "post_content", "old": "Latest news", "new": "Latest News & Updates" }
     ]
   }
   ```
6. **Repeat** for remaining open annotations. The user transitions `done → verified` (pin disappears) or `done → open` (sends it back with their feedback) via the toolbar.

### Authentication

All endpoints require `manage_options`. Pick whichever auth fits the agent:

- **Logged-in cookie** — agents running in a wp-admin session inherit cookie + `X-WP-Nonce`
- **Application Password** — for remote agents, HTTP Basic auth with a per-user app password
- **JWT/OAuth plugin** — Bearer token works the same way

### Push Notifications

Configure under Tools → Annotate AI:
- **Webhook:** POSTs batch JSON with optional Bearer token auth. URL must be `http://` or `https://`. Treat as a privileged setting — no SSRF allow-list beyond scheme.
- **Telegram:** Sends formatted message via Bot API. Token format `\d+:[A-Za-z0-9_-]+`, chat ID is numeric. Message body is HTML-escaped (`parse_mode: HTML`).

## REST API

All endpoints require `manage_options` capability.

| Method | Endpoint | Description |
|--------|----------|-------------|
| `GET` | `/annotate-ai/v1/annotations` | List (filter: `?status=open`, `?page_url=...`) |
| `POST` | `/annotate-ai/v1/annotations` | Create single annotation. Used by the toolbar in None mode (live save). |
| `POST` | `/annotate-ai/v1/annotations/batch` | Batch submit + notify agent. Used by the toolbar in webhook/telegram mode. |
| `PATCH` | `/annotate-ai/v1/annotations/{id}` | Update note, requested_text, and/or requested_changes on an existing annotation. Used by edits in None mode. |
| `POST` | `/annotate-ai/v1/annotations/{id}/resolve` | Resolve with note (stored as `resolution_note`) |
| `DELETE` | `/annotate-ai/v1/annotations/resolved` | Clear resolved |
| `POST` | `/annotate-ai/v1/settings` | Save settings (validates Telegram token, chat ID, webhook URL scheme) |

`requested_changes` is server-sanitised against an allow-list (`font-size`, `color`, `background-color` for v1). Keys outside the allow-list are dropped — extend the list in `Annotate_AI::sanitize_requested_changes()` when the modal grows new controls.

## Version Bumping

Always bump the version in **three places**:
1. Plugin header in `annotate-ai.php` (the `Version:` line)
2. The `ANNOTATE_AI_VERSION` constant in `annotate-ai.php`
3. The `version` field in `package.json`

WordPress uses the version string for browser cache busting on frontend assets.

## Accessibility

Most a11y comes from `@wordpress/components` for free:
- `<Modal>` provides `role="dialog"`, `aria-modal`, `aria-labelledby`, focus trap, Escape handling, focus restoration
- `<Snackbar>` provides `role="status"` and `aria-live="polite"`
- `<Notice status="error">` provides `role="alert"`
- `<Button>` provides focus-visible rings, hover states, disabled handling
- All components honour `prefers-reduced-motion` internally

Plugin-specific additions:
- Toolbar toggle button uses `aria-pressed` to expose state
- Numbered pins have descriptive `aria-label` (`"Edit annotation N"`)
- The selected-element selector display has an `aai-sr-only` label so screen readers announce its role

## What NOT to Do

- Don't reintroduce custom modal/snackbar/button/control implementations — use the matching `@wordpress/components` exports so the toolbar evolves with WordPress
- Don't use a CPT for storage — options are intentional
- Don't add priority or category fields — they were deliberately removed
- Don't show the toolbar to non-admins
- Don't hardcode colors, spacing, or radii — use `@wordpress/base-styles` tokens
- Don't trust the client's `user` field — derive from `wp_get_current_user()` server-side
- Don't add untranslated user-facing strings — wrap them in `__()` and update the `.pot`/`.po`/`.mo`/`.json` files

## Testing

### Automated

```bash
npm run check-types     # tsc --noEmit
npm run lint            # ESLint + Stylelint + lint-pkg-json
npm run test:unit       # Jest unit tests for src/frontend/utils.ts
composer test           # PHPUnit tests for sanitisers + build_annotation
                        # (or: phpunit -c phpunit.xml.dist)
```

The PHP tests stub the small set of WordPress functions our sanitisers depend on (`sanitize_text_field`, `sanitize_textarea_field`, etc.) so they run standalone without booting WP — fast feedback on the security-relevant paths.

### Manual smoke test

Run this end-to-end after any non-trivial change. Should take ~5 minutes.

**Setup**
- [ ] Plugin activated on a block-theme site (block theme exercises the preset path)
- [ ] Logged in as `manage_options` user, admin bar visible

**Toggle + admin-bar UX**
- [ ] On the frontend, the **Annotate** item appears in the admin bar with the dashicon
- [ ] Click it → bar item gains the focus/active style; toast says "Annotation mode on…"
- [ ] Press Esc → toast says "Annotation mode off"
- [ ] Hide the admin bar in user profile → reload → floating "Annotate" button appears bottom-right

**Annotate (None mode)**
- [ ] Settings → Annotate AI → method = None. Pull URL + agent snippet visible. **Copy URL** and **Copy agent snippet** flip to "Copied" briefly
- [ ] Activate annotation mode → hover an element (theme-color outline appears) → click it
- [ ] Modal opens with the element's selector, current text (if text-only), font size, colours pre-filled
- [ ] Bump font size to a preset size → pick a colour from the palette → add a note → click **Add**
- [ ] Pin appears (blue), toast says "Annotation saved"
- [ ] Reload the page — pin is still there, still blue (persistent pin)
- [ ] Click the pin → modal pre-fills with what you saved, including the preset
- [ ] Change something → **Update** → toast "Annotation updated"

**Status flow / agent simulation**
- [ ] Get the annotation's `id` from `GET /wp-json/annotate-ai/v1/annotations`
- [ ] `PATCH /wp-json/annotate-ai/v1/annotations/{id}` with `{"status":"in_progress"}` → reload → pin is yellow, click is disabled
- [ ] PATCH with `{"status":"done","resolution_note":"Bumped size","changes":[{"file":"theme.json","path":"...","old":"18px","new":"24px"}]}` → reload → pin is green
- [ ] Click the green pin → review modal shows your request and the agent's resolution + change list
- [ ] **Looks good** → pin disappears
- [ ] Re-create another → mark done → click → **Not quite** → pin returns to blue (open) and is editable again

**Webhook / Telegram modes**
- [ ] Settings → method = Webhook → set URL + token → save (invalid token format → 400 with translated message)
- [ ] Annotate two elements → "Send 2 annotations" pill appears bottom-right → click → toast "2 annotations sent to agent"
- [ ] Pins remain visible after send (now have server IDs)

**Breakpoints**
- [ ] In the modal, change **Affects** to "Mobile only" → save → GET the annotation; `breakpoint` is `"mobile"`

**Theme isolation / accessibility**
- [ ] On a theme that styles bare `button { transform: scale(1.05) }`, hover the modal's primary button — no scale leak, font-family is WP default
- [ ] Tab cycles inside the modal only; Esc closes; focus restores to the trigger
- [ ] Screen reader announces the modal title; the "Selected element:" prefix is announced before the selector

**i18n**
- [ ] Settings → General → Site Language → French → reload — admin page and toolbar both translated
- [ ] Switch to Japanese — confirm same
- [ ] Switch back to English

**Color scheme**
- [ ] Users → Profile → Admin Color Scheme → pick a non-default scheme — the toolbar buttons on the frontend follow the chosen accent color
